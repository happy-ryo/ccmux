ccmux のステータスバーに出る「🤖 evaluator · 🔧 Bash · ▓▓░░ 3/8 · 45.2k tokens」は、**Claude Code が吐いているローカルの JSONL を読んで**作っています。公開 API を叩いているわけではなく、ユーザーのマシン内で完結する情報です。

このファイルの場所・構造・気をつけるポイントを整理します。

## ファイルの場所

Claude Code は会話ログを次の場所に書き出します:

```
~/.claude/projects/<encoded-cwd>/<uuid>.jsonl
```

`encoded-cwd` は**現在の作業ディレクトリのパスを、英数字以外すべて `-` に置換**したものです。例えば `C:\Users\じゅぶ\Desktop\dev\ccmux` は:

```
C--Users-----Desktop-dev-ccmux
```

日本語の 3 文字 `じゅぶ` が `---` に潰れているのがミソで、初期の実装では ASCII 英数字だけを保持して他を取り除いていたため、日本語パスのユーザーだけ JSONL が見つからない不具合がありました。

```rust title="src/claude_monitor.rs"
fn encode_cwd_to_project_name(cwd: &Path) -> String {
    cwd.to_string_lossy()
        .chars()
        .map(|c| if c.is_ascii_alphanumeric() { c } else { '-' })
        .collect()
}
```

## JSONL の中身

各行は 1 イベント。抜粋するとこんな形:

```json
{"type":"assistant","message":{"content":[{"type":"tool_use","id":"toolu_01...","name":"Bash","input":{"command":"ls"}}]},"requestId":"req_01...","usage":{"input_tokens":12,"cache_read_input_tokens":15234,"output_tokens":8}}
{"type":"user","message":{"content":[{"type":"tool_result","tool_use_id":"toolu_01...","content":"README.md\nsrc\n..."}]}}
{"type":"assistant","message":{"content":[{"type":"tool_use","name":"Agent","input":{"subagent_type":"evaluator","prompt":"..."}}]}}
```

ccmux が拾っているフィールドは主に:

- **`type` + `message.content[].type`**: `tool_use` / `tool_result` / `text`
- **`tool_use.name`**: 現在実行中のツール名（Bash / Edit / Read / Agent など）
- **`tool_use.input.subagent_type`**: `Agent` ツール経由で spawn したサブエージェントの種類
- **`usage.*`**: トークン消費量
- **`message.stop_reason`**: `tool_use` なら作業継続中、`end_turn`/`null` なら完了

## 二重計上を避ける

トークン計算でハマったのが、**同じ `requestId` のレコードが 3 回続けて現れる**こと。Claude Code は 1 つのターンに対して複数エントリを書き、そのそれぞれが同じ usage を持っています。単純に足すとトークン消費が 3 倍に見える。

```rust
if let Some(req_id) = extract_request_id(&event) {
    if !self.counted_request_ids.insert(req_id.to_string()) {
        return; // already counted this request
    }
}
self.state.input_tokens += usage.input_tokens;
// ...
```

`HashSet<String>` で `requestId` を覚えておき、二回目以降をスキップ。

## ストリーミング読み取り

Claude Code は JSONL にどんどん追記していくので、ccmux は:

1. `file_position` を保持
2. `mtime` を見て、変わってたらファイルを開いて前回位置から読む
3. ファイルサイズが前回より小さくなってたら truncation と判断し、position をリセット
4. 500ms ごとにポーリング（短すぎると IO が無駄、長すぎるとラグが出る）

あと、5 秒ごとに**プロジェクトディレクトリ全体を再スキャン**して新しい JSONL（＝ 新しい会話セッション）が現れていないかチェックしています。

## ロック戦略

複数ペインで同時に JSONL を読むと `Arc<Mutex<ClaudeMonitor>>` の競合が起きがち。そこで 3 段階の短いロックに分解しました:

1. **短ロック**: 次に読むべきパスと `file_position` を取る
2. **ロックなし**: ファイル IO（これが一番重い）
3. **短ロック**: 読んだイベントを state に反映

こうすると長時間ロックが走らないので、描画スレッドがブロックされて UI がカクつく現象が消えます。

---

というわけで、**Claude Code は JSONL を公開情報として書き出してくれている**ので、それを丁寧に読むだけで色んな情報が拾えます。次回は `stop_reason` の取扱いと、サブエージェントの lifecycle 追跡について書きます。
